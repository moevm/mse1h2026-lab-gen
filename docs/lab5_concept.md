# Концепция лр №5 "регулярные выражения"
## Общее описание
Лабораторная работа посвящена обработке текстовых данных с использованием регулярных выражений.

Студенту необходимо разработать программу, которая анализирует текст, содержащий различные типы технических данных, извлекает из него записи, удовлетворяющие заданным условиям.

На вход программе подается текст, представляющий собой набор строк. Текст заканчивается строкой `Fin.`. В тексте могут встречаться различные структуры в зависимости от варианта задания: логи терминала Linux, URL-ссылки, IP-адреса, финансовые записи или серверные логи. Гарантируется, что если строка содержит какой-то пример корректной для варианта структуры, то он полностью располагается в пределах одной строки. В рамках конкретного варианта корректной считается только структура, удовлетворяющая всем параметрам и условиям варианта.

Лабораторная работа разделяется на два этапа:
- Этап 1: Проверка корректности структуры:
	На данном этапе студенту выдается одна из задач с параметризацией. Задача студента состоит в нахождении подстрок, соответствующих данному формату. На этом этапе для каждого пункта параметризации студенту назначается ровно один вариант из перечисленных.
- Этап 2: Проверка специальных условий:
	После нахождения на этапе 1 всех корректных структур студент должен дополнительно отфильтровать их по условиям фильтрации. На этом этапе студенту может быть выдано несколько условий фильтрации из перечисленных. Их количество можно настроить в зависимости от сложности.

Результатом работы программы являются:
1. количество структур, успешно прошедших этап 1 (проверку корректности структуры);
2. все структуры, прошедшие этап 2 (условия фильтрации).
На первой строке необходимо вывести количество структур, успешно прошедших этап 1.
Далее необходимо вывести все структуры, удовлетворяющие условиям этапа 2, каждую с новой строки, в порядке появления во входных данных.
Если после этапа 2 подходящие структуры отсутствуют, необходимо вывести:
`Empty`
Если на этапе 1 корректные структуры отсутствуют, программа должна вывести:
```
0
Empty
```

Разрешается использовать несколько регулярных выражений.

Каждому студенту назначается одна основная задача. У каждой задачи присутствуют параметры для персонализации 1 этапа и условия для фильтрации на 2 этапе.

Значения параметров и условий определяются автоматически в зависимости от сида студента.

## Задачи
### 1. Логи терминала
Команда в терминале имеет следующий формат:
`<username>@<hostname>:~<symbol> <command>`
- `<username>` — имя пользователя: латинские буквы, цифры и символ `_`, при этом первым символом может идти только латинская буква;
- `<hostname>` — имя компьютера: латинские буквы, цифры и символы `_`, `-`; при этом первым символом может идти только латинская буква;
- `<symbol>`:
    - `$` — пользовательская оболочка;
    - `#` — оболочка суперпользователя;
- `<command>` — текст команды, может содержать - латинские буквы, цифры, пробелы, символы `.`, `/`, `-`, `_`, `|`, `>`, `<`.
#### Параметризация:
1) Наличие флагов в command
	- Содержит/не содержит флаги вообще
	- Содержит/не содержит полные флаги (начинаются с `--`)
	- Содержит/не содержит краткие флаги (начинаются с `-`)
	- Содержит любое количество флагов
2) Наличие путей в command
	- Содержит хотя бы один абсолютный путь
	- Содержит хотя бы один относительный путь
	- Не содержит путей
	- Может содержать пути любого вида
3) Формат username
	- Обязан содержать символ `symb` (определяется вариантом)
	- Не должен содержать символ `symb` (определяется вариантом)
4) Формат hostname
	- Обязан содержать символ `symb` (определяется вариантом)
	- Не должен содержать символ `symb` (определяется вариантом)
#### Условия фильтрации:
1) Только оболочка root `#`/user `$`
2) command содержит pipe `|`
3) command содержит перенаправление (`>` или `<`)
4) command начинается с заданного слова (gcc, docker, python, git и тд)
5) hostname содержит `-`
6) hostname содержит/не содержит число
7) username содержит/не содержит число

### 2. Ссылки
Формат URL:
```<protocol><domain><port><path><query><fragment>```
где:
- `<protocol>` — протокол (`http`, `https`, `ftp`) + `://`;
- `<domain>` — доменное имя;
- `<port>` — порт в формате `:N`, где `N` — число от 1 до 65535;
- `<path>` — путь, начинающийся с символа `/`;
- `<query>` — query-параметры, начинающиеся с символа `?`;
- `<fragment>` — фрагмент, начинающийся с символа `#`.
Поля `<port>`, `<path>`, `<query>` и `<fragment>` могут отсутствовать в зависимости от параметров варианта. Порядок элементов фиксирован.
#### Параметризация:
1) Наличие протокола:
	- обязательно должен быть
	- быть не должно
	- может быть, а может нет
2) Наличие URL-пути
	- должен быть обязательно
	- быть не должно
	- может быть, а может нет
3) Наличие query
	- должен быть обязательно
	- быть не должно
	- может быть, а может нет
4) Наличие порта
	- должен быть любой
	- должен быть из интервала от N до M
	- быть не должно
	- может быть, а может нет
#### Условия фильтрации:
1) Домен должен содержать заданную доменную зону (.ru, .com, .org, .edu)
2) path содержит файл с заданным расширением
3) path содержит не менее/не более N сегментов
4) query содержит параметр с заданным именем
5) query содержит не более/не менее N параметров 

### 3. IP-адреса
Формат IP-адресов:
ipv4: X.X.X.X, где X - число от 0 до 255
#### Параметризация:
1) ведущие нули
	- разрешены
	- запрещены
2) адреса с маской подсети по нотации CIDR (например 192.168.0.1/20)
	- разрешены
	- запрещены
3) адреса с портами (например 192.168.0.1:8080)
	- разрешены
	- запрещены
4) адреса в шестнадцатеричном формате (XX.XX.XX.XX, где X - 16-ричное число)
	- разрешены только с заглавными буквами
	- разрешены только со строчными буквами
	- разрешены любые
	- запрещены
#### Условия фильтрации:
1) адрес содержит/не содержит повторяющиеся октеты
2) маска подсети не меньше/не больше N (в случае, если маски разрешены)
3) порт принадлежит заданному диапазону (в случае, если порты разрешены)
4) только loopback адреса
5) только multicast адреса

### 4. Финансовые данные
Формат операций:
`<datetime> <type> <amount> <currency>`
- datetime - дата и время в формате: `[YYYY-MM-DD HH:MM:SS]`(параметризация может поменять разделители)
- type - тип операции (TRANSFER, WITHDRAW, DEPOSIT, PAYMENT)
- amount - количество денег, число без ведущих нулей, дробная часть разделяется `,`. Не может содержать более 2-х знаков после запятой.
- currency - валюта, 3 буквы латинского алфавита
#### Параметризация:
1) Дробные числа в amount
	- только целые числа
	- только дробные числа
	- допустимы и дробные, и целые числа
2) Разделители тысяч
	- тысячи делятся `.`
	- тысячи делятся `_`
	- тысячи разделителей нет
	- может быть любой из разделителей `.`, `_` или отсутствовать вообще
3) Формат даты
	- день, месяц и год в дате разделяются `/`
	- день, месяц и год в дате разделяются `.`
	- день, месяц и год в дате разделяются `-`
4) Формат валюты
	- только заглавные буквы
	- только строчные буквы
	- без учёта регистра
#### Условия фильтрации:
1) type только определённая операция
2) amount больше/меньше N
3) amount принадлежит заданному диапазону
4) currency только заданная валюта
5) операции только в заданном диапазоне времени

### 5. Серверные логи
Формат логов:
`<datetime> <level> <service> <message>`
- datetime - дата и время в формате: `[YYYY-MM-DD HH:MM:SS]`(параметризация может поменять разделители или обособление)
- level - уровень важности(INFO, DEBUG, WARN, ERROR, CRITICAL)
- service - имя компонента/сервиса, состоит из латинских букв, цифр и символов `-`, `_`, `.`
- message - сообщение
#### Параметризация:
1) Обособление datetime
	- обособлена скобками `[]`
	- обособлена скобками `{}`
	- обособлена скобками `<>`
	- обособлена любым видом скобок `[]`, `<>`, `{}`
	- никак не обособлена
2) Формат даты
	- день, месяц и год в дате разделяются `/`
	- день, месяц и год в дате разделяются `.`
	- день, месяц и год в дате разделяются `-`
3) Обособление level
	- обособлена скобками `[]`
	- обособлена скобками `{}`
	- обособлена скобками `<>`
	- обособлена любым видом скобок
	- никак не обособлена
4) Разделитель между элементами
	- элементы разделены пробелом
	- элементы разделены `|`
	- элементы разделены `;`
#### Условия фильтрации:
1) только записи заданного уровня level
2) логи только в заданном диапазоне времени
3) service содержит `-`
4) в service все буквенные символы только верхнего/нижнего регистра
5) message содержит/не содержит число
6) message содержит указанное слово (timeout, failed, denied, exception и тд)

### 6. Логи Apache
Формат логов Apache:

`[<datetime>] "<method> <path> <protocol>" <status> <size>`
где:
- `<datetime>` — дата и время запроса в формате: `[YYYY-MM-DD HH:MM:SS]`(параметризация может поменять разделители или обособление)
- `<method>` — HTTP-метод (`GET`, `POST`, `PUT`, `DELETE`, `PATCH`)
- `<path>` — путь к ресурсу
- `<protocol>` — версия HTTP-протокола (`HTTP/1.0`, `HTTP/1.1`, `HTTP/2`)
- `<status>` — HTTP-код ответа
- `<size>` — размер ответа в байтах
Пример: `[12/May/2025:14:22:01] "GET /index.html HTTP/1.1" 200 5120`
#### Параметризация
1) Обособление datetime
	- обособлена скобками `[]`
	- обособлена скобками `{}`
	- обособлена скобками `<>`
	- обособлена любым видом скобок `[]`, `<>`, `{}`
	- никак не обособлена
2) Формат даты
	- день, месяц и год в дате разделяются `/`
	- день, месяц и год в дате разделяются `.`
	- день, месяц и год в дате разделяются `-`
3) Формат path
	- path обязан оканчиваться расширением
	- path не должен содержать расширение
	- path может содержать query-параметры
	- path не содержит query-параметры
4) Обособление status
	- обособлена скобками `[]`
	- обособлена скобками `{}`
	- обособлена скобками `<>`
	- обособлена любым видом скобок
	- никак не обособлена
#### Условия фильтрации:
1) только запросы заданного метода (GET, POST и тд)
2) только запросы со status определённого класса (2xx, 4xx и тд)
3) size в определённом диапазоне
4)  только заданный протокол (`HTTP/1.0`, `HTTP/1.1`, `HTTP/2`)
5) логи только в заданном диапазоне времени

## Пример варианта
### Задача: Серверные логи
На вход программе подается текст, представляющий собой набор строк. Текст заканчивается строкой `Fin.`. В тексте могут встречаться серверные логи. Кроме корректных структур, текст может содержать произвольные данные, поврежденные записи и конструкции, частично похожие на корректные.
Требуется, используя регулярные выражения, найти все корректные серверные логи, удовлетворяющие условиям варианта.
Гарантируется, что если строка содержит корректный лог, то он полностью располагается в пределах одной строки.
Обработка должна выполняться в два этапа:
1. поиск структур, удовлетворяющих формату и параметрам этапа 1;
2. фильтрация найденных структур по условиям этапа 2.

Логи имеют следующий формат:
```
<datetime> <level> <service> <message>
```
где:
- `datetime` — дата и время;
- `level` — уровень важности (`INFO`, `DEBUG`, `WARN`, `ERROR`, `CRITICAL`);
- `service` — имя компонента/сервиса;
- `message` — сообщение.

**Параметры этапа 1**
- datetime обособлен скобками `[]`;
- level обособлен скобками `<>`;
- компоненты даты разделяются `-`;
- разделитель между элементами — `|`.

**Условия фильтрации этапа 2**
- level должен быть `ERROR`;
- service должен содержать символ `-`.

**Формат вывода**
На первой строке необходимо вывести количество логов, успешно прошедших этап 1.
Далее необходимо вывести все логи, удовлетворяющие условиям этапа 2, каждый с новой строки.
Если после этапа 2 подходящие записи отсутствуют, необходимо вывести:
`Empty`
**Входные данные**
```
[2025-05-12 12:10:44]|<INFO>|auth-service|user login successful
[2025-05-12 12:15:01]|<ERROR>|auth-service|request timeout detectedbroken log line
[2025-05-12 12:16:02]|<CRITICAL>|dbservice|timeout while reading data
[2025-05-12 12:17:55]|<ERROR>|auth-handler|connection timeout
2025-05-12 12:18:01|ERROR|auth-service|timeoutFin.
```
**Результат**
```
4
[2025-05-12 12:15:01]|<ERROR>|auth-service|request timeout detected
[2025-05-12 12:17:55]|<ERROR>|auth-handler|connection timeout
```
