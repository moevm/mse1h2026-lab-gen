# Генератор лабораторных работ по программированию

## О проекте

Проект посвящён разработке **генератора персонализированных заданий** для лабораторных работ по программированию.

Ключевая идея: каждому студенту назначается уникальный, но воспроизводимый вариант задания на основе его ФИО (через детерминированный seed/hash). Это позволяет:
- повысить разнообразие заданий;
- сохранить тематику каждой лабораторной работы;
- упростить проверку и повторное восстановление варианта;
- снизить риск плагиата решений.

Результат проекта - генераторы заданий для 7 лабораторных работ, интегрируемые в курс на e.moevm.info

---

## Проблема, которую решает проект

В существующих лабораторных работах по программированию часто возникают следующие проблемы:

- **однотипность формулировок** - у многих студентов задания совпадают или отличаются незначительно;
- **сложность масштабирования** - преподавателю трудно вручную готовить большое число качественных вариантов;
- **невоспроизводимость** - при повторной проверке бывает неудобно точно восстановить ранее выданный вариант;
- **риск плагиата** - одинаковые постановки провоцируют копирование решений;
- **неодинаковая пригодность к “рандомизации”** - часть ЛР нельзя просто параметризовать числами, нужна переработка постановки.

Проект решает это за счёт:
- параметризованных шаблонов;
- набора концепций вариативности для каждой ЛР;
- детерминированной генерации параметров и структуры варианта по seed/hash студента.

---

## Цель проекта

Разработать генераторы заданий для 7 лабораторных работ, которые:

- сохраняют тематику каждой ЛР;
- дают персонализированную вариативность;
- воспроизводят один и тот же вариант по ФИО (и номеру ЛР);
- могут быть встроены в курс.

## Постановка задачи

Необходимо разработать решение для генерации персонализированных заданий для лабораторных работ по программированию со следующими требованиями:

1. **Добавить разнообразие** в существующие задания (всего **7 ЛР**).
2. Для **каждой ЛР** предложить **по 3 концепции** вариативности, сохранив тематику работы:
   - набор тем и навыков должен оставаться тем же;
   - часть работ нельзя просто “зарандомить”, для таких работ требуется переработка постановки и проектирование нескольких концепций.
3. Обеспечить **воспроизводимость “уникального” задания**:
   - в формулировке задания есть **N настраиваемых параметров**;
   - значение **N** определяется отдельно для каждой ЛР на этапе проектирования концепции;
   - от ФИО студента вычисляется seed/hash;
   - seed/hash преобразуется во внутренней функции в конкретные значения параметров;
   - значения параметров зависят **только от ФИО студента** (и выбранной ЛР / концепции), а не от внешних факторов.

---

## Ключевая идея решения

Общий принцип генерации:

1. Выбирается лабораторная работа (и при необходимости концепция).
2. Вводится ФИО студента.
3. ФИО нормализуется.
4. По ФИО и идентификатору ЛР вычисляется seed/hash.
5. Из seed/hash детерминированно вычисляются:
   - структура варианта (например, какие подзадачи назначаются),
   - значения параметров.
6. Параметры подставляются в шаблон формулировки.
7. Получается готовое персонализированное задание.

## Функциональные требования

### Обязательные
- Генерация задания для выбранной ЛР по ФИО студента.
- Поддержка параметризованных концепций генерации.
- Поддержка настраиваемого количества параметров `N` для конкретной ЛР/концепции.
- Поддержка детерминированного выбора структуры варианта по seed/hash.
- Формирование итоговой формулировки задания на основе шаблона.
- Воспроизводимость результата при одинаковом вводе.

### Расширяемые
- Поддержка нескольких концепций (по 3) для каждой ЛР.
- Валидация входных данных (формат ФИО).
- Защита от невалидных комбинаций параметров.
- Отладочный режим (нормализованное ФИО, seed/hash, параметры).
- Вывод в формате, удобном для Moodle / CodeRunner.

---

## Нефункциональные требования

- **Воспроизводимость** - одинаковый ввод должен давать одинаковый результат.
- **Предсказуемость** - отсутствие зависимости от времени и окружения.
- **Расширяемость** - добавление новых ЛР/концепций без переписывания ядра.
- **Поддерживаемость** - читаемые правила генерации и документированные шаблоны.
- **Надёжность** - отсутствие нерешаемых/некорректных вариантов при корректной настройке.
- **Интегрируемость** - возможность использования в Moodle / CodeRunner.
- **Прозрачность для проверки** - возможность восстановить структуру варианта и параметры.

## Сценарии использования (Use Cases)

### UC-1. Генерация персонализированного задания по ФИО
**Пользователь:** Преподаватель / ассистент  
**Цель:** Получить индивидуальный вариант задания для выбранной ЛР.

**Предусловия:**
- Для выбранной ЛР настроена хотя бы одна концепция генерации.
- Определены шаблон и правила параметризации.

**Сценарий:**
1. Пользователь выбирает ЛР.
2. Пользователь вводит ФИО студента.
3. Система нормализует ФИО.
4. Система вычисляет seed/hash.
5. Система детерминированно выбирает структуру варианта (если предусмотрено концепцией).
6. Система вычисляет значения параметров.
7. Система подставляет параметры в шаблон задания.
8. Система возвращает готовую формулировку.

**Результат:**
- Сформирован воспроизводимый персонализированный вариант задания.

---

### UC-2. Повторное воспроизведение варианта для проверки
**Пользователь:** Проверяющий / преподаватель  
**Цель:** Восстановить ранее выданное задание студента.

**Предусловия:**
- Известны ЛР и ФИО студента.
- Используется та же версия правил генерации.

**Сценарий:**
1. Пользователь выбирает ЛР.
2. Вводит ФИО студента.
3. Система повторяет алгоритм генерации.
4. Система выдаёт тот же вариант (структуру и параметры).

**Результат:**
- Вариант совпадает с ранее выданным.

---

### UC-3. Проектирование концепции генерации для ЛР
**Пользователь:** Преподаватель / разработчик  
**Цель:** Подготовить концепцию вариативности для конкретной ЛР.

**Сценарий:**
1. Определяется учебная цель ЛР.
2. Выделяются элементы задания, которые можно параметризовать.
3. Определяется число параметров и/или структура варианта.
4. Задаются диапазоны и ограничения.
5. Описывается шаблон формулировки.
6. Проверяется генерация на тестовом наборе ФИО.

**Результат:**
- Концепция генерации готова к реализации и использованию.

---

### UC-4. Получение задания студентом
**Пользователь:** Студент  
**Цель:** Получить свой персонализированный вариант ЛР.

**Предусловия:**
- Генератор интегрирован в курс или используется преподавателем вручную.

**Сценарий:**
1. Студент открывает задание в курсе / получает формулировку от преподавателя.
2. Студент видит свой вариант с назначенными параметрами.
3. Выполняет ЛР согласно постановке.

**Результат:**
- Студент получает индивидуальное задание в рамках темы ЛР.
